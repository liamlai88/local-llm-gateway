# Multi-Agent 混合架构突破报告

> **核心问题**：单一架构总有局限——纯 LLM Multi-Agent 慢且贵，纯规则 Multi-Agent 快但盲区多。能否把两者组合？  
> **本实验**：实现"规则快路径 + LLM 兜底慢路径"的混合架构，让系统**自动判断**走哪条路。  
> **结论**：**准确率 4/4 等同 Plan-Execute Turbo，平均延迟仅 1/8（1.2s vs 9.8s）。命中题毫秒级，盲区题 LLM 兜底**。  
> **实验时间**：2026.04

---

## 架构演进背景

之前 5 份 Agent 实验揭示了单一架构的局限：

| 架构 | 优势 | 致命弱点 |
|------|------|----------|
| ReAct | 简单 | 多工具任务 25% 天花板 |
| Plan-Execute | 准确率高 | 延迟 5-30s，每次都调多次 LLM |
| 纯规则 Multi-Agent | **0.6ms 极速** | 盲区无法处理 |

**真实生产场景**：80% 的请求是高频简单问题，20% 是低频复杂问题。**用同一种架构处理两类问题都是浪费**。

## 混合架构设计

### 整体流程

```
                  用户问题
                     │
          ┌──────────▼──────────┐
          │  CoordinatorAgent    │ (规则路由)
          │  关键词 + 正则识别    │
          └──────────┬──────────┘
                     │
          ┌──────────▼──────────────────────┐
          │   Worker Agents 并行执行         │
          │   ├── RetrieverAgent (RAG)      │
          │   ├── WeatherAgent              │
          │   ├── CalculatorAgent           │
          │   └── SolutionAgent             │
          └──────────┬──────────────────────┘
                     │
          ┌──────────▼──────────┐
          │   CriticAgent        │ (审计层)
          │   - 必跑的跑了吗？   │
          │   - Solution 是空话？│ ← 关键判断
          │   - KB 有内容但没用？│
          └──────────┬──────────┘
                     │
              passed?│
              ┌──────┴──────┐
              │             │
            YES│           NO│
              │             ▼
              │     ┌─────────────────┐
              │     │ LLMFallbackAgent│  ⭐ 慢路径
              │     │ 用 Qwen-Turbo   │
              │     │ + KB context    │
              │     │ 兜底回答         │
              │     └────────┬────────┘
              │              │
              ▼              ▼
        ┌─────────────────────┐
        │   FinalizerAgent     │ (合成最终答案)
        └──────────────────────┘
```

### 关键创新：Critic 升级为质量裁判

**v1 Critic**（旧）：只检查"必跑的有没有跑"
```python
if needs_calculator:
    check(calculations非空)
```

**v2 Critic**（新）⭐：检查"答案有没有内容"
```python
# 新增 1: Solution 空话检测
if "未识别到" in advice or "建议先补充" in advice:
    fail("SolutionAgent 输出为空话")

# 新增 2: KB 有内容但规则路径未利用
if kb_has_product_content and plan["product"] is None:
    fail("KB 有产品文档但规则未识别 → 需 LLM 兜底")
```

**这是混合架构的灵魂**：Critic 不是只看流程完整性，而是**质量裁判**，决定要不要升级到 LLM 兜底。

## 实验结果

### 4 道测试题（2 命中 + 2 盲区）

| 测试 | 命中规则路由的关键词？ | Plan-Execute Turbo | 纯规则 MA | **混合 MA** |
|------|------------------------|---------------------|----------|-------------|
| 杭州/北京气温差 | ✅ 城市名+气温差 | ✅ 3680ms | **✅ 0.6ms** | **✅ 0.6ms** (fast) |
| A100 一天多少钱 | ✅ 产品名+时长 | ✅ 6500ms | ✅ 400ms | **✅ 400ms** (fast) |
| 70B 大模型推荐 | ❌ 无具体产品名 | ✅ 12521ms | ❌ 334ms | **✅ 2576ms** (slow) |
| 推理选型咨询 | ❌ 无产品+无价格关键词 | ✅ 15519ms | ❌ 假通过 | **✅ 1867ms** (slow) |

### 三种架构总览

| 架构 | 准确率 | 平均延迟 | 单次成本 | 命中区延迟 | 盲区延迟 |
|------|--------|----------|----------|------------|----------|
| Plan-Execute Turbo | **4/4** | 9790ms | 4× LLM 调用 | 5-7s | 12-15s |
| 纯规则 Multi-Agent | 3/4 | **267ms** ⚡ | **0** | <1s | <1s 但不准 |
| **混合 Multi-Agent** ⭐ | **4/4** | **1202ms** | **0~1× LLM** | **<1s** | **2-3s** |

## 三个核心洞察

### 洞察 1: 混合架构 = 准确率上限 × 延迟下限

```
单一架构选择题 (旧思维):
  要准 → Plan-Execute Turbo (慢, 9.8s)
  要快 → 纯规则 Multi-Agent (盲区不准)

混合架构 (新思维):
  ✅ 准确率 = Plan-Execute Turbo (4/4)
  ✅ 延迟 = Plan-Execute Turbo / 8 (1.2s 平均)
  
  关键: 80% 请求走快路径，20% 走 LLM 兜底
        加权平均后，性价比远超任何单一架构
```

### 洞察 2: Critic 是混合架构的"路由开关"

混合架构的成败 100% 取决于 Critic 的判断准确度：

```
Critic 太宽松 → 漏判，盲区题被错误判为成功 (假成功)
Critic 太严格 → 误判，简单题被升级到 LLM (浪费)
Critic 精准   → 完美路由 (我们的目标)
```

**升级 Critic 的两个关键检测**：
1. **空话检测**：识别 "未识别到"、"建议先补充" 这类兜底话
2. **资源利用检测**：KB 有内容但规则路径没用上 → 必须升级

### 洞察 3: LLM 兜底必须用强模型

```python
# 错误设计
LLM_Fallback 用 1.5B 本地模型
→ 兜底也答不好 (1.5B 在开放推理上能力不足)

# 正确设计 ⭐
LLM_Fallback 硬编码 Qwen-Turbo
→ 兜底就要用强模型，因为已经触发兜底说明任务复杂
```

**类比**：医院的"专家会诊"必须叫主任医师，不能再叫住院医师。

## 与 Plan-Execute Turbo 的本质区别

表面看两个架构都能 4/4 通过，但**底层经济模型完全不同**：

```
Plan-Execute Turbo (旧):
  每个请求都调 4-6 次 LLM
  无论问题复杂度
  → 成本与请求数线性增长

混合 Multi-Agent (新):
  80% 请求 0 次 LLM 调用
  20% 请求 1 次 LLM 调用
  → 成本与"复杂请求数"线性，与"总请求数"次线性

实际部署 (10 万次/天):
  Plan-Execute Turbo: 10 万 × 4 次 LLM = 40 万次 → ~¥800/天
  混合 Multi-Agent:    8 万 × 0 + 2 万 × 1 = 2 万次 LLM → ~¥40/天
  
  ⭐ 成本节省 95%
```

## 完整 9 份实验报告认知曲线

```
1. Prompt Engineering ROI    → 模型容量是地板
2. RAG vs 纯 LLM            → 闭域 0% → 100%
3. Hybrid Search 失败        → 检索没有银弹
4. Rerank 突破              → 三层架构是标配
5. ReAct Agent 边界          → Agent 不是万能
6. Plan-Execute 突破         → 范式 > 模型规模
7. MCP Server 实现           → 工具集 USB 化
8. LoRA 微调突破            → 100 条样本 = 小模型完胜大模型
9. Multi-Agent 混合架构 ⭐   → 规则 + LLM 组合 = 准确率 ↑ 成本 ↓ 95%
```

完整覆盖 **Prompt → RAG → Agent → 协议 → 微调 → 架构** 的 GenAI 工程全栈。
